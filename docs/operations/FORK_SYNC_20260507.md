## 为什么用 cherry-pick 而不是 rebase

| 方式 | 适用场景 | 不适用原因 |
|------|---------|-----------|
| **rebase** | 本地无自定义提交，纯同步上游 | ❌ 本地有大量自定义提交（ADS MCP、数据采集等），rebase 会重放所有本地历史，每次都要重新解决冲突 |
| **cherry-pick** | 保留双方提交历史 | ✅ 只有上游新提交需要逐个 cherry-pick，本地提交不受影响，冲突只出现在上游改过且本地也改过的文件 |

**结论**：对于 Fork 仓库（有大量自定义功能），cherry-pick 比 rebase 更可控、冲突更少。

## cherry-pick 流程

```bash
# 1. 获取上游最新代码
git fetch upstream

# 2. 找出 fork 分叉点之后上游的所有提交
#    （从 backup-main 分支的 HEAD 开始算）
git log --oneline upstream/main --not backup-main

# 3. 逐条 cherry-pick（遇到冲突则解决后继续）
git cherry-pick <commit-hash>

# 4. 合并提交处理（带 -m 1 指定主父提交）
git cherry-pick -m 1 <merge-commit-hash>
```

## 同步后修复的 Bug

### Bug 1：502 Bad Gateway — Next.js SSR 挂死

| 项目 | 内容 |
|------|------|
| **症状** | `http://localhost:2026/` 正常，`/workspace` 等 SSR 路由返回 502 |
| **根因** | frontend 使用 `target: dev` + `pnpm run dev` 触发 Next.js 16.1.7 + Turbopack 在 Docker Alpine 环境下 SSR 时挂死 |
| **修复** | 改为 `target: prod` + `pnpm start`（使用预编译产物，不依赖 dev server） |
| **涉及文件** | `docker/docker-compose-dev.yaml` |

### Bug 2：502 Bad Gateway — nginx 解析失败

| 项目 | 内容 |
|------|------|
| **症状** | nginx 日志显示 `frontend could not be resolved (2: Server failure)` |
| **根因** | frontend 容器启动后立即崩溃重启，`command` 中 `pnpm start` 在 `/app` 目录执行（需在 `/app/frontend` 目录） |
| **修复** | `command` 改为 `cd frontend && pnpm start > /app/logs/frontend.log 2>&1` |
| **涉及文件** | `docker/docker-compose-dev.yaml` |

### Bug 3：Cross-site auth request denied

| 项目 | 内容 |
|------|------|
| **症状** | 注册时返回 `403 Cross-site auth request denied` |
| **根因** | CSRF 中间件校验 `Origin` 时需要 `X-Forwarded-Host` 头计算请求源，但 nginx `/api/` 块未转发该头 |
| **修复** | 在 nginx.conf 的 `/api/` location 块添加 `proxy_set_header X-Forwarded-Host $host:$server_port` |
| **涉及文件** | `docker/nginx/nginx.conf` |

### Bug 4：nginx 重定向地址错误

| 项目 | 内容 |
|------|------|
| **症状** | Next.js SSR redirect 返回 `308`，`Location` 指向容器内部地址 `http://frontend:3000/...` |
| **根因** | nginx 未覆盖 upstream 返回的 `Location` 头 |
| **修复** | 在 `/` location 块添加 `proxy_redirect http://$frontend_upstream http://$host:$server_port` |
| **涉及文件** | `docker/nginx/nginx.conf` |

### Bug 5：api-client.ts 重复代码

| 项目 | 内容 |
|------|------|
| **症状** | `pnpm build` 报错 `the name 'readCsrfCookie' is defined multiple times` |
| **根因** | cherry-pick 冲突残留，导入语句和函数定义出现了两遍 |
| **修复** | 删除重复行 |
| **涉及文件** | `frontend/src/core/api/api-client.ts` |
