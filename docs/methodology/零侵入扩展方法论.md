# 零侵入扩展方法论

## 核心理念

所有自定义功能优先放在扩展目录，绝不直接修改 DeerFlow 官方源码。只有当扩展目录无法实现时，才允许在核心源码打补丁——且必须 `try/except` 包裹、幂等注入、**完整记录到 `docs/`**。

## 三条铁律

### 铁律一：扩展目录优先

**能放扩展目录的逻辑，绝不改核心源码。**

检查顺序：
1. 功能能否完全在 `frontend/extensions/` 或 `deerflow_extensions/` 内实现？
2. 如果能——所有代码放在扩展目录，零侵入核心
3. 如果不能——能否通过已有注入点对接（如 SettingsDialog registry）？
4. 如果还不能——才考虑在核心源码打最小补丁

### 铁律二：补丁必须可逆

**必须改核心源码时，必须做到三点：**
1. `try/except ImportError` 包裹所有扩展模块的 import 和调用——扩展不可用时 DeerFlow 仍能正常运行
2. 全局 `_installed` 变量（或等效守卫）保证注入逻辑幂等
3. 在 `docs/patches/` 下完整记录：文件路径、行号、diff、原因、风险等级

### 铁律三：改造完成必须归档

**改造完成后必须归档文档到 `docs/`——扩展 README、补丁记录、变更记录缺一不可。**

具体归档要求见下方"文档归档规范"。

## 三层扩展模式

从低侵入到高侵入分为三个 Level，每层有对应的文档归档要求：

| Level | 扩展方式 | 核心源码改动 | 必须创建的文档 |
|-------|---------|-------------|-------------|
| Level 1 | 纯扩展 | 无 | 扩展目录 `README.md` |
| Level 2 | 注入扩展 | try/except 注入 | 同上 + `docs/patches/` + `docs/changelog/code_change_summary/` |
| Level 3 | 补丁扩展 | 少量源码改动 | 同上 + `docs/patches/` + `docs/changelog/code_change_summary/` |

### Level 1：纯扩展（零侵入）

扩展目录完整实现功能，核心源码无需任何修改。

| 端 | 实现方式 | 实际案例 |
|----|---------|---------|
| 前端 | 独立 React 组件，在页面/布局中直接 import 使用 | `mobile-sidebar` — 独立组件，页面中引入 |
| 前端 | Registry 注册表模式，运行时注册 | `input-suggestions` — `registerInputSuggestion()` 注册 |
| 后端 | 独立 FastAPI router，通过 startup.py 注入 | `env_settings` — 独立 router，startup.py 中 `app.include_router()` |

**文档归档要求：** 扩展目录下创建 `README.md`（说明、目录结构、核心文件职责、使用方式）。

### Level 2：注入扩展（低侵入）

扩展目录实现全部逻辑，核心源码通过已有注入点对接。

核心源码改动模式：
```python
# 在核心源码的启动块中
try:
    from deerflow_extensions.xxx.startup import install_xxx
    install_xxx()
except ImportError:
    pass
```

| 端 | 注入点 | 实际案例 |
|----|--------|---------|
| 前端 | `core/settings-extensions/registry.ts` | `env-settings` — 注册为 SettingsDialog 配置页 |
| 后端 | `app/gateway/app.py` 的 `# Extension Hooks` 区块 | `env_settings` — `install_env_settings()` |

**文档归档要求：**
1. 扩展目录下创建 `README.md`
2. 核心源码改动记录到 `docs/patches/` 下对应模块文件
3. 变更记录到 `docs/changelog/code_change_summary/` 下对应模块文件

### Level 3：补丁扩展（中侵入）

扩展目录实现业务逻辑，核心源码需少量非注入式修改（如中间件守卫）。

| 端 | 补丁形式 | 实际案例 |
|----|---------|---------|
| 前端 | middleware.ts 追加判断逻辑 | `ads_auth` — 路由重写 + Cookie 检查 |
| 后端 | middleware dispatch 加 1 行守卫 | `ads_auth` — `auth_middleware.py` 的 Extension Hook |
| 后端 | monkey-patch 代理方法 | `data_collection` — 替换 `_build_middlewares` |

**文档归档要求：**
1. 扩展目录下创建 `README.md`
2. 补丁详细记录到 `docs/patches/` 下对应模块文件
3. 变更记录到 `docs/changelog/code_change_summary/` 下对应模块文件

## 决策树（含文档归档）

```
新功能需求
    │
    ├─ 能否完全放在扩展目录？
    │   ├─ 能 → Level 1：纯扩展 ✅
    │   │      ├─ 实现代码
    │   │      ├─ 扩展目录下创建 README.md
    │   │      └─ （可选项）如需详细 API 文档，在 docs/ 下按模块建子目录
    │   │
    │   └─ 不能 → 能否通过已有注入点对接？
    │        ├─ 能 → Level 2：注入扩展 ✅
    │        │      ├─ 实现代码
    │        │      ├─ 扩展目录下创建 README.md
    │        │      ├─ 核心源码改动记录到 docs/patches/
    │        │      └─ 变更记录到 docs/changelog/code_change_summary/
    │        │
    │        └─ 不能 → Level 3：补丁扩展 ⚠️
    │               ├─ 实现代码
    │               ├─ 扩展目录下创建 README.md
    │               ├─ 核心源码补丁记录到 docs/patches/
    │               └─ 变更记录到 docs/changelog/code_change_summary/
```

## 文档归档规范

| 文档类型 | 要求 | 参考 |
|---------|------|------|
| 扩展 README | 每个扩展目录必须有，说明功能、目录结构、核心文件职责、使用方式、依赖 | `frontend/extensions/ads_auth/README.md` |
| API 文档（可选） | 当扩展有对外 API 时创建，记录接口签名、参数、返回值 | `deerflow_extensions/data_collection/API.md` |

| 变更记录 | 所有核心源码改动必须记录到 `docs/changelog/code_change_summary/` | `docs/changelog/code_change_summary/` |

## 补丁契约

当必须修改核心源码时，需遵循以下契约：

### 1. 代码要求

```python
# ✅ 正确：try/except 包裹 + _installed 守卫
_installed = False

def install_xxx(app=None):
    global _installed
    if _installed:
        return
    try:
        if app is None:
            from app.gateway.app import app as _app
            app = _app
        from deerflow_extensions.xxx import router
        app.include_router(router)
        _installed = True
    except Exception as e:
        import logging
        logging.getLogger(__name__).warning("[Xxx] install failed: %s", e)
```

### 2. 文档要求

在 `docs/patches/` 下的对应模块文件中记录：

```markdown
### 补丁名称

- **文件**: `app/gateway/xxx.py`
- **行号**: 42-45
- **改动**: 在 `xxx` 函数中追加 3 行守卫
- **原因**: 对接扩展模块 xxx 的认证注入
- **风险**: ✅ 低（try/except 包裹，扩展不可用时走原逻辑）
```

### 3. 同步后检查

从 ByteDance 上游 `git pull` 后，必须检查 `docs/patches/README.md` 中记录的所有补丁是否被覆盖。
