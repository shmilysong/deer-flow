## 一、前提：DeerFlow 的业务场景画像

在分析前，先明确我们要蒸馏的业务场景：

| 维度 | 说明 |
|------|------|
| **业务类型** | 企业智能客服 + 内部业务 Agent（查终端、执行运维操作、知识库问答等） |
| **教师模型** | DeepSeek API（公有云） |
| **蒸馏方式** | 黑盒指令蒸馏（仅用 API 输出文本，无 Logits 访问权） |
| **关键能力** | 工具调用（MCP）、RAG 知识问答、多轮对话、合规拒绝 |
| **部署环境** | 企业内部服务器，私有化部署，数据不出内网 |
| **性能要求** | 首 token 延迟 < 1s，并发 ≥ 10 req/s，输出质量 ≥ 教师模型 90% |

## 三、🥉 青铜方案：Qwen3-8B（入门级）

### 3.1 适合场景

- **单部门智能客服**：日均 100-500 次会话
- **轻量 Agent 任务**：RAG 问答 + 简单工具调用（1-2 个 MCP 工具）
- **预算敏感**：希望在消费级硬件上跑推理

### 3.2 硬件需求

#### 训练（QLoRA 4bit）
| 配置 | 规格 |
|------|------|
| GPU | RTX 4090 24GB × 1  |
| CPU | 8 核以上 |
| 内存 | 32GB |
| 硬盘 | 100GB SSD |
| 训练时间 | 6-12 小时（1万条数据，约 3 epoch） |

#### 推理（vLLM 部署）
| 配置 | 规格 |
|------|------|
| GPU | RTX 4090 24GB × 1（或 RTX 3090 / A10） |
| FP16 显存占用 | ~16 GB（权重）+ ~4 GB（KV cache）= **~20 GB** |
| INT8 显存占用 | ~10 GB（权重）+ ~4 GB（KV cache）= **~14 GB** |
| 首 token 延迟 | ~200ms（FP16） / ~300ms（INT8） |
| 最大并发 | ~8 req/s（单卡） |
| 推理吞吐 | ~40 tok/s（输出） |

### 3.3 数据量需求

| 数据阶段 | 最少 | 推荐 | 最优 |
|---------|------|------|------|
| 原始采集点 | 3,000条 | 30,000条 | 100,000条 |
| 清洗后聚合样本 | 1,000条 | 10,000条 | 30,000条 |
| 蒸馏训练数据 | 500条 | 5,000条 | 15,000条 |

### 3.4 效果预期

| 评估维度 | 基准（DeepSeek API） | 8B 蒸馏后 | 相对水平 |
|---------|-------------------|----------|---------|
| RAG 知识问答准确率 | 95% | 78-85% | **~85%** |
| 工具调用（MCP）成功率 | 98% | 80-88% | **~85%** |
| 多轮对话连贯性 | 优秀 | 良好 | **~85%** |
| 合规拒绝准确率 | 99% | 90-95% | **~95%** |
| 幻觉率 | <1% | 3-5% | **可控** |
| 推理延迟（P95） | ~800ms（公网） | ~300ms（本地） | **快 60%** |

### 3.5 业界参考案例

- **Agent Distillation 论文（2025 ACL）**：0.5B/1.5B/3B 模型经 Agent 蒸馏后，在 HotPotQA、MATH 等任务上，仅用 **~2,000 条轨迹** 就达到跨档竞争力
- **DistilQwen2.5-0.5B**：仅 0.5B 参数量，蒸馏后在 42.81% 基准上达到未蒸馏 7B 模型水平
- **结论**：8B 模型在 **5,000 条** 高质量 Agent 数据上即可显著有效

### 3.6 优缺点

| ✅ 优势 | ❌ 不足 |
|---------|---------|
| 显卡成本 < 3万 | 复杂多步推理能力不足（3+ 工具链） |
| 可部署在消费级 PC | 对罕见业务场景泛化差 |
| 推理速度快 | 高精度需求场景（如财务计算）有风险 |
| 训练门槛低 | 幻觉率相对较高 |

## 四、🥈 白银方案：Qwen3-14B（主流级·推荐）

### 4.1 适合场景

- **企业级智能客服**：日均 500-5,000 次会话
- **通用业务 Agent**：RAG 知识库 + 3-5 个 MCP 工具调用
- **中等复杂任务**：多步推理（2-3步）、多工具串联
- **性价比最优**：推荐作为首档目标

### 4.2 硬件需求

#### 训练（QLoRA 4bit）
| 配置 | 规格 |
|------|------|
| GPU | A100 40GB × 1 **或** RTX 4090 24GB × 2 **或** L40S 48GB × 1 |
| CPU | 16 核以上 |
| 内存 | 64GB |
| 硬盘 | 200GB SSD |
| 训练时间 | 12-24 小时（1万条数据，约 3 epoch） |

⚠️ **注意**：24GB 单卡（RTX 4090）**装不下 14B QLoRA 训练**（需要 ~28GB），最少需要 2 张 4090 或用 48GB 的 L40S / A40。推荐用 **L40S 48GB 单卡**，显存刚好够，无多卡通信开销。

#### 推理（vLLM 部署）
| 配置 | 规格（FP8/AWQ） |
|------|----------------|
| GPU | L40S 48GB × 1 **或** RTX 4090 24GB × 2（TP=2） |
| FP8 显存占用 | ~14 GB（权重）+ ~6 GB（KV cache）= **~20 GB** ← 24GB 单卡勉强可跑 |
| INT4/AWQ 显存占用 | ~7 GB（权重）+ ~6 GB（KV cache）= **~13 GB** ← 24GB 单卡轻松跑 |
| 首 token 延迟 | ~350ms（FP8） / ~500ms（INT4） |
| 最大并发 | ~12 req/s |
| 推理吞吐 | ~52 tok/s（单卡，FP8） |

### 4.3 数据量需求

| 数据阶段 | 最少 | 推荐 | 最优 |

| 原始采集点 | 10,000条 | 50,000条 | 200,000条 |
| 清洗后聚合样本 | 3,000条 | 15,000条 | 50,000条 |
| 蒸馏训练数据 | 1,500条 | 8,000条 | 30,000条 |

|---------|-------------------|-----------|---------|
| RAG 知识问答准确率 | 95% | 88-93% | **~93%** |
| 工具调用（MCP）成功率 | 98% | 90-95% | **~95%** |
| 多轮对话连贯性 | 优秀 | 优秀 | **~95%** |
| 合规拒绝准确率 | 99% | 95-98% | **~97%** |
| 多步推理正确率（3步+） | 92% | 82-88% | **~90%** |

### 4.5 业界参考案例

- **DeepSeek-R1 官方蒸馏**：用 R1 蒸馏的 14B 模型（DeepSeek-R1-Distill-Qwen-14B），在数学推理上 **超越 QwQ-32B-Preview**，证明 14B 模型经过高质量蒸馏可以达到极大模型水平

## 七、数据量-效果关系深度分析（业界实证）

### 7.1 核心发现：数据量不是越多越好

根据阿里 DistilQwen 大规模蒸馏实验的关键数据：

```
效果提升 vs 数据量 关系曲线
───────────────────────────────────────────

效果
↑ 100% ──────────────── 饱和区
↑  95%        ╱         (>100K 条，增益递减)
↑  90%    ╱──          黄金区
↑  80%  ╱               (10K-100K 条，提升最明显)
↑  60%╱                 起步区
     └────┴────┴────┴──→ 数据量
        1K   10K  100K 1M
```

| 数据量区间 | 效果特征 | 说明 |
|-----------|---------|------|
| **~1K 条** | 有效果但有限 | 模型学到基本模式，但泛化差 |
| **~5K 条** | 大部分场景可工作 | 覆盖常见模式，简单场景可用 |
| **10K-100K 条** | 🏆 **黄金区间** | 收益最明显，DistilQwen 实验证实 |
| **>100K 条** | 增益递减 | 需要更多样化而非更多数据 |
| **1M+ 条** | 边际效应明显 | DeepSeek 蒸馏 R1 用 800K，但那是**通用模型** |

### 7.2 质量 > 数量

> **"100 条高质量轨迹 > 10,000 条噪音数据"**

| 数据质量维度 | 重要性 | 说明 |
|------------|--------|------|
| **轨迹完整性** | ⭐⭐⭐⭐⭐ | 必须有完整的 user → thought → tool_call → tool_result → response 链路 |
| **难度覆盖** | ⭐⭐⭐⭐ | 简单/中等/困难比例 3:5:2 |
| **场景多样性** | ⭐⭐⭐⭐ | 覆盖所有 MCP 工具和业务场景 |
| **正确率** | ⭐⭐⭐⭐ | 错误工具调用/错误回复必须过滤 |
| **数据去重** | ⭐⭐⭐ | 基于 (query + response hash) 精确去重 |

### 7.3 对于 DeerFlow 的具体数据规划

| 场景类别 | 需要条数 | 采集周期（日均1万原始） | 备注 |
|---------|---------|----------------------|------|
| RAG 知识问答 | 3,000-5,000 | 3-5天 | 最常见场景，数据易得 |
| Agent 工具调用 | 2,000-3,000 | 3-5天 | 需覆盖 5-10 种 MCP 工具 |
| 多轮复杂任务 | 1,000-2,000 | 5-10天 | 需要多轮会话，采集慢 |
| 合规拒绝 | 500-1,000 | 持续累积 | 稀有样本，需人工标注补充 |
| **总计（8B/14B 起步）** | **8,000-15,000** | **~2周** | 足够启动首次蒸馏训练 |

### 9.1 推理成本对比（vs 公网 API）

## 十一、总结与推荐

### 核心结论

| 问题 | 答案 |
|------|------|
| **需要多少数据？** | 5,000-15,000 条高质量聚合样本可启动，30,000-50,000 条可达接近教师水平 |
| **选什么模型？** | **🥈 14B 性价比最优**，32B 为终极目标 |
| **什么服务器？** | 起步 RTX 4090（￥1.5万），推荐 L40S（￥4万），终极 H100（￥25万） |
| **达到什么效果？** | 14B→教师 93% 能力，32B→教师 98% 能力，且推理速度快 50-60% |
| **多久能上线？** | 8B 验证：2周；14B 主力：4-6周；32B 终极：8-16周 |
| **能省多少钱？** | 年化节省 API 调用费 **￥6-18万**，1-2 年收回硬件投资 |
