# DeerFlow 数据飞轮与数据中心蓝图设计

> **WING** 2026-06-02
>
> 本文件为 DeerFlow 智能客服 Agent 数据飞轮及未来企业级数据中心平台提供整体架构蓝图。
> 仅作设计参考，不涉及具体代码实现。

## 一、背景与目标

### 1.1 现状

- **DeerFlow** 已作为公司智能客服引擎部署到客户内网（学校/单位）
- 集成了 **ADS 业务 MCP**（47+ 工具）和 **DeepRAG 知识库 MCP**
- **data\_collection 扩展**已投产，6 个采集点（P1-P6）覆盖全链路 LLM 推理数据
- 已有 **ETL 清洗聚合流水线**和 **用户反馈系统**（±1 赞踩）
- 日均 <1K 会话，模型使用公有 API（DeepSeek）

### 1.2 核心目标

| 目标         | 说明                              |
| ---------- | ------------------------------- |
| **客服质量提升** | 自动化评估 + Bad Case 发现 + 人工审阅确认的闭环 |
| **模型蒸馏降本** | 高质量数据积累 → QLoRA 蒸馏 → 灰度替代公有 API |
| **知识库自优化** | 识别知识缺口，生成候选条目，建议人工更新            |
| **统一数据中心** | 所有数据源统一上报、统一清洗、统一存储分析           |

### 1.3 设计原则

| 原则         | 说明                                 |
| ---------- | ---------------------------------- |
| **零侵入**    | DeerFlow 端不修改核心代码，所有功能走扩展目录        |
| **先报告后行动** | 所有改动必须经人工审核确认，AI 只出报告和建议           |
| **渐进演进**   | 先从 Agent 数据飞轮开始，再逐步扩展为通用数据中心       |
| **不重复造轮子** | 充分利用 RuoYi-Vue-Plus 已有能力，只开发业务差异部分 |
| **数据单向上报** | 客户内网数据只出不进，数据中心不反向访问客户网络           |

## 二、RuoYi 能力复用策略

数据中心基于 RuoYi-Vue-Plus 搭建，其大量通用能力可直接复用，无需重新开发。

### 2.1 RuoYi 内置能力与数据中心需求的映射

| RuoYi 内置能力            | 复用到数据中心               | 需要开发吗？    |
| --------------------- | --------------------- | --------- |
| **用户管理 + 权限体系**       | 系统用户登录、角色权限、按钮级鉴权     | ❌ 直接用     |
| **部门 / 岗位管理**         | 映射为数据中心的团队/组织架构       | ❌ 直接用     |
| **菜单管理**              | 控制数据看板各页面的访问权限        | ❌ 直接用     |
| **操作日志**              | 数据中心所有人工操作的审计追溯       | ❌ 直接用     |
| **登录日志**              | 用户登录行为追踪              | ❌ 直接用     |
| **定时任务 (Quartz)**     | 管线调度——ETL/评分/报告等定时触发  | ⚡ 只需配任务定义 |
| **代码生成器**             | 基础 CRUD 页面（字典管理、配置管理） | ❌ 直接用     |
| **文件管理**              | 文件上传存储基础设施            | ❌ 直接用     |
| **通知公告**              | 主子决策提醒、异常告警推送         | ❌ 直接用     |
| **字典管理**              | 数据分级枚举、Bad Case 分类枚举  | ❌ 直接用     |
| **参数配置**              | 评分权重、管线阈值等动态配置        | ❌ 直接用     |
| **Swagger / Knife4j** | API 文档自动生成            | ❌ 直接用     |
| **数据权限**              | 不同角色看不同数据源的数据         | ❌ 直接用     |

### 2.3 定时任务复用说明

RuoYi 内置的 Quartz 定时任务直接承担管线调度职能：

```
Quartz 任务调度
  ├─ daily_etl               每日 03:00  各数据源 ETL 清洗
  ├─ daily_scoring            每日 03:30  质量评分
  ├─ daily_report             每日 04:00  质量报告生成
  ├─ weekly_knowledge_gap     每周一 04:00 知识缺口识别
  └─ weekly_report            每周一 05:00 周报生成
```

每个定时任务→调用对应 Service→触发管线编排。无需引入额外调度组件。

### 3.2 各层开发类型总览

| 层次                | 开发类型         | 说明                                                     |

## 四、Agent 数据飞轮闭环（一期核心）

### 4.1 三层飞轮节奏

```
L1: 质量监控闭环（天级）
┌───────────────────────────────────────────────┐
│  部署点上报 → 数据中心评分 → 生成质量日报       │
│  → 主子审阅确认 → Bad Case 标记 → 决策归档     │
│  （只出报告，不自动改动任何配置）                │
└───────────────────────────────────────────────┘

L2: 知识建议闭环（周级）
┌───────────────────────────────────────────────┐
│  Bad Case 归因分析 → 识别知识缺口              │
│  → AI 生成候选知识条目 → 主子审阅             │
│  → 主子自行更新知识库                          │
└───────────────────────────────────────────────┘

L3: 蒸馏迭代闭环（月级）
┌───────────────────────────────────────────────┐
│  高质量数据池构建 → 主子确认启动蒸馏训练        │
│  → LlamaFactory QLoRA → 生成蒸馏模型          │
│  → A/B 灰度测试 → 主子审批全量上线             │
│  → 逐步减少公有 API 调用 → 降本                │
└───────────────────────────────────────────────┘
```

### 4.2 高质量数据池设计（L3 基础）

LLM 自动评分维度（每维度 1-5 分）：

| 维度       | 满分含义          | 权重  |
| -------- | ------------- | --- |
| **完整性**  | 全面覆盖用户问题，无遗漏  | 25% |
| **准确性**  | 事实正确，工具结果准确反映 | 30% |
| **工具使用** | 工具选择正确、参数合理   | 20% |
| **有用性**  | 直接解决用户实际需求    | 15% |
| **语言质量** | 表达清晰、专业、流畅    | 10% |

质量分级：

| 等级     | 复合分范围    | 数据池用途         |
| ------ | -------- | ------------- |
| **铂金** | ≥4.5     | 蒸馏训练首选，采样权重最高 |
| **黄金** | 3.5\~4.5 | 蒸馏训练主力        |
| **白银** | 2.5\~3.5 | 辅助训练          |
| **青铜** | <2.5     | 不入训练集，仅归档分析   |

### 4.3 关键设计决策

| 决策         | 选择                        | 理由                         |
| ---------- | ------------------------- | -------------------------- |
| **数据上报方式** | HTTPS API 主动推送            | 客户内网能出公网但公网进不来，只能由部署端主动推   |
| **评分模型**   | GPT-4o-mini（使用公司 API Key） | 评分质量可靠、成本极低，不和用户共享 API 配额  |
| **评分时机**   | 上报后公司侧离线执行                | 凌晨一次性批量处理，不影响生产环境          |
| **高质量数据池** | 5 维评分 + 用户反馈融合 + 四档分级     | 参考 LMSys/Chatbot Arena 方法论 |
| **改动审批**   | 所有改动需主子在 Web 看板确认         | 安全可控，不信任全自动 AI             |
| **调度引擎**   | 复用 RuoYi Quartz 定时任务      | 无需引入额外调度组件，配置即用            |

### 6.1 管线与调度

### 6.2 AI 引擎集成方式

AI 引擎（Python FastAPI）定位为 Java 后端的"计算加速器"，不直接对外。

| ---------- | --------------------------------- | ----------------------------------------- |

## 七、数据清洗策略

### 7.1 核心原则：清洗 ≠ 丢弃

数据清洗不是简单的"脏数据扔掉"。一条数据是否"脏"，取决于**用在什么地方**：

| 数据用途            | 处理方式             | 原因                 |
| --------------- | ---------------- | ------------------ |
| **蒸馏训练集**       | 排除低质量/错误数据       | 训练有毒数据会降低模型效果      |
| **Bad Case 分析** | **错误数据是最有价值的输入** | 错误暴露系统弱点，是改进的起点    |
| **质量报告**        | 全量统计分析           | 需要完整的数据才能算出真实的质量分布 |

> **关键认知**：一条包含工具调用错误的对话，对训练是"垃圾"，对系统改进是"黄金"。
> 飞轮的核心价值正是从这些错误中发现问题、驱动改进。

### 7.2 部署端预清洗（轻量、全自动）

在客户内网部署点上凌晨 03:00 由 cron 触发，处理后即刻上报。

**不会丢弃的错误数据**：

| 数据类型             | 处理方式                    | 流向                                 |
| ---------------- | ----------------------- | ---------------------------------- |
| **工具调用返回 error** | 保留，标记 `tool_error=true` | 上报 → Bad Case 管线分析工具失败模式           |
| **模型输出截断**       | 保留，标记 `truncated=true`  | 上报 → 分析是否需要调整 max\_tokens          |
| **模型拒绝回答**       | 保留，标记 `rejected=true`   | 上报 → 分析是否 topic\_guardrail 误杀/知识缺口 |
| **多轮对话未完成**      | 保留，标记 `incomplete=true` | 上报 → 统计中断率                         |

**真正需要丢弃的只有一种**：缺少关键字段（session\_id / user\_query / 模型回复全部为空）的无意义空记录。

**确定性规则处理（不需 AI）**：

| 步骤        | 规则                                        | 目的                     |
| --------- | ----------------------------------------- | ---------------------- |
| **去重**    | 基于 user\_query + model\_response 的 MD5 去重 | 避免用户重复提问产生的冗余          |
| **标记短回复** | 模型回复 < 5 字符不丢弃，只打标签 `short_reply=true`    | 保留完整性，由公司端分析决定         |
| **标记错误**  | 含 error 字段的记录不丢弃，只打标签 `has_error=true`    | 错误数据是 Bad Case 分析的核心素材 |

**设计考量**：部署端只做标记（tagging），不做丢弃（filtering）。真正的分流决策留到公司端由 AI 辅助判断。标记后的全量数据上报，确保公司端能拿到完整的画面。

### 7.3 公司端分流与深度处理（管线化、可配置）

数据上报到数据中心后，根据标记将数据分流到不同路径：

```
                    上报数据（全量，含各类标记）
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        has_error     short_reply    正常数据
        rejected      truncated
              │            │            │
              ▼            ▼            ▼
       ┌──────────┐ ┌──────────┐ ┌──────────┐
       │ Bad Case  │ │ 质量分析  │ │ 评分管线  │
       │ 分析管线   │ │ 管线      │ │          │
       └──────────┘ └──────────┘ └────┬─────┘
              │            │           │
              ▼            ▼           ▼
         工具失败模式   截断原因分析   5维评分
         知识缺口识别   拒绝原因分析   分级入库
         错误趋势统计   短回复模式     数据池
```

| 深度处理步骤      | 规则                                                             | 执行方                    |
| ----------- | -------------------------------------------------------------- | ---------------------- |
| **格式校验**    | 验证 JSONL 格式、Messages Schema 合规性                                | AI Engine (Python)     |
| **异常检测**    | 统计方法识别：异常长的回复、异常多的工具调用、异常高的 token 消耗 → 标记后进入 Bad Case          | AI Engine (pandas)     |
| **多轮完整性校验** | 检查 tool\_calls 与 tool\_results 是否一一对应 → 不完整则进入 Bad Case        | AI Engine              |
| **质量预标记**   | 对明显的低质量样本（重复话术、截断回复、空工具结果）预置低分 → 进入质量分析                        | AI Engine              |
| **数据分流路由**  | 根据标签将数据分发到对应管线：正常数据 → 评分管线 / 错误数据 → Bad Case 管线 / 低质量 → 质量分析管线 | PipelineService (Java) |

**重要**：当前代码 `clean_and_aggregate.py` 中的 `filter_error_cases` 方法直接丢弃了错误数据，这是设计缺陷。正式实现时应改为**分流标记**而非**丢弃过滤**。

### 7.4 数据分流总览图

```
部署端 (03:00 cron)
  ETL 脚本
    └─ MD5 去重 → 多标签标记 (has_error/short_reply/rejected/truncated/incomplete)
    └─ 输出: train_data.jsonl (全量，每条带标签)
    └─ HTTPS 上报（全量，不丢弃错误数据）

公司端 (接收到数据后 Quartz 触发)
  PipelineService:
    ├─ 读取标签 → 按标签分流
    │
    ├─ 正常数据 → AI Engine 5 维评分 → 分级入库 → 数据池
    │
    ├─ 错误数据 → AI Engine Bad Case 分析
    │     ├─ 工具失败模式统计 (哪个工具/哪种参数/什么错误)
    │     ├─ 拒绝原因归类 (合规拒绝/知识不足/理解错误)
    │     └─ 输出: Bad Case 报告 + 知识缺口建议
    │
    ├─ 短回复/截断 → AI Engine 质量分析
    │     ├─ 统计截断率/短回复率
    │     └─ 输出: 系统参数调优建议
    │
    └─ 异常数据 → 直接入 Bad Case 记录
```

### 8.2 并发场景与处理策略

数据中心需要处理的并发场景分为两类：**实时请求**和**批量任务**，两者互不干扰。

#### 实时请求：数据上报（多部署点同时推送）

```
部署点 A ──→ HTTPS POST ──┐
部署点 B ──→ HTTPS POST ──┼──→ Nginx ──→ Tomcat 线程池 ──→ 接收层 Controller
部署点 C ──→ HTTPS POST ──┘
```

| 层              | 并发策略                                                                                   |
| -------------- | -------------------------------------------------------------------------------------- |
| **Nginx**      | `worker_connections` 默认 1024，多部署点同时上报不会互相阻塞；可通过 `limit_req_zone` 配置单 IP 限流，防止单个部署点过度请求 |
| **Tomcat 线程池** | Spring Boot 默认 200 线程，每个上报请求占用 1 个线程，200 个并发上报绰绰有余；文件接收采用 Streaming 模式，不将整个文件加载到内存     |
| **文件存储**       | 接收层将文件流式写入 MinIO 的 `/raw/{source_code}/{date}/` 目录，不同部署点写不同路径，无文件锁竞争                   |
| **数据库写入**      | HikariCP 连接池（默认 10 连接）承载上报记录登记，写入操作以 `source_code + date` 为粒度，冲突概率极低                   |

**并发承载能力**：以当前日均 <1K 会话、聚合后每次上报 \~20MB 计，即使 50 个部署点同时在凌晨 3:10 上报，单机轻松应对。

#### 批量任务：管线处理（凌晨 Quartz 调度）

```
Quartz 定时任务 (03:00-05:00 串行触发)
  ├─ ETL 管线      → AI Engine（同步 HTTP 调用）
  ├─ 评分管线       → AI Engine（批量调用 LLM，逐条评分）
  ├─ 入库管线       → PostgreSQL（批量 INSERT）
  ├─ Bad Case 管线  → AI Engine（分析已评分数据）
  └─ 报告管线       → AI Engine（生成报告文件）
```

| 层             | 并发策略                                                                                      |
| ------------- | ----------------------------------------------------------------------------------------- |
| **Quartz 任务** | 管线任务配置为 **串行执行**（不允许同一数据源的同类型管线并发跑），避免同时写 MinIO 或评分结果覆盖；不同数据源的管线可并行                       |
| **LLM 评分**    | AI Engine 内部将 200-500 条样本分批（每批 20 条），逐批调用 GPT-4o-mini API，批间串行、批内并行，兼顾 API rate limit 和吞吐 |
| **数据库批量写**    | 评分结果/入库采用 MyBatis-Plus 的 `saveBatch` 批量插入，每批 100 条，减少数据库往返                                |
| **管线锁**       | Redis 分布式锁（`dc:pipeline:{source_code}:{type}:lock`），防止 Quartz 任务异常重入                      |

### 8.3 并发架构关键原则

| 原则             | 说明                                                                     |
| -------------- | ---------------------------------------------------------------------- |
| **读写分离**       | 上报写 MinIO → 管线读 MinIO → 结果写 PostgreSQL → 看板读 PostgreSQL，各环节操作不同目标，无锁竞争 |
| **同步收、异步算**    | 数据接收立即返回（同步），管线处理离线进行（异步），用户上报体验不受管线影响                                 |
| **串行管线、并行数据源** | 同一数据源的管线串行执行（保证数据一致性），不同数据源的管线可并行（互不影响）                                |
| **流量可控**       | 日均 <1K 会话，并发压力极低；即使未来扩展到 100+ 部署点，部署端分散在凌晨 3:00-4:00 窗口上报，天然错峰         |

## 十二、关键原则

1. **改动必审**：所有影响线上行为的操作必须经过人工审核确认。AI 只出报告和建议，不改动任何配置。
2. **零侵入**：对已部署的 DeerFlow 服务不做任何核心代码改动，所有采集功能通过扩展目录实现。
3. **不重复造轮子**：RuoYi 已有的用户/权限/日志/调度/代码生成等能力直接使用；业务模块的 CRUD 代码由代码生成器生成骨架，只开发差异化的业务逻辑。
4. **渐进式架构**：数据中心为多数据源设计，但一期只实现 Agent 飞轮。以数据源扩展机制为骨架，后续新增数据源只需注册 + 配置管线，无需修改核心系统。
5. **数据单向上报**：客户内网数据只出不进，数据中心不反向访问客户网络，保障客户安全合规。
6. **管线编排而非硬编码**：数据处理的每个环节（ETL/评分/入库/报告）都是独立可编排的管线单元，组合方式由配置决定而非代码。
